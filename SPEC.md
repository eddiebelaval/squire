# SPEC

## Identity

Squire is a file-based toolkit that changes how AI coding agents behave. It ships as a collection of markdown files and shell scripts that you drop into your project or global config directory. The core artifact is `squire.md`, a behavioral ruleset that addresses the 9 most common failure patterns observed across 5 months of intensive AI-augmented development. Around that core sit 63 slash commands, 328 skills, 34 custom agents, an 11-stage build pipeline, 7 behavioral patterns, 6 thinking frameworks, the Triad documentation system, and workspace generators for file-based project organization.

## Current Capabilities

### 1. Behavioral Ruleset

- **squire.md**: 9 rules derived from 3,307 commits across 5 months. Each rule targets a measured failure mode: planning instead of building, pushing without preflight, dismissing visual bugs, scoping changes too broadly, batching edits without verification, over-engineering, environment mismatches, and two more.
- **Drop-in activation**: Copy to project root or `~/.claude/CLAUDE.md` and agent behavior changes immediately.
- **Patterns directory**: Extended behavioral guidance in `patterns/behavioral-rules.md`.

### 2. Slash Commands

- **63 commands** covering shipping (/ship), deployment (/deploy), research (/research), visualization (/visualize), planning (/blueprint), testing (/test), publishing (/publish), and operations (/morning, /status).
- **Format**: Each command is a markdown prompt file in `commands/`.
- **Agent-tool-native**: Commands instruct agents to use their built-in tools systematically, not to call external services.

### 3. Skills Library

- **328 skills** across 17 domains: frontend, backend, architecture, DevOps, AI/ML, SEO, CRO, content, marketing, financial, startup, operations, product management, design, UX, writing, and communication.
- **Format**: Markdown files in `skills/`, each providing domain expertise as agent context.

### 4. Custom Agents

- **34 agents** with tool access for architecture, security, DevOps, payment processing, database migrations, MCP servers, social media, market intelligence, and more.
- **Format**: Markdown files in `agents/`, each defining an executable specialist (not a static prompt).

### 5. Stage-Gate Pipeline

- **11 stages**: Concept Lock through Listen & Iterate.
- **Gate enforcement**: Each stage has a gate question that must be answered before advancing.
- **Agent-native additions**: AI-specific workflow steps at each stage.
- **Branch hygiene**: Rules for branch naming, commit conventions, and merge discipline.
- **Format**: Markdown files in `pipeline/`.

### 6. Thinking Frameworks and Patterns

- **7 behavioral patterns** in `patterns/`: metacognitive awareness, PEV (plan-execute-verify), inversion, reflection, tree of thoughts, ensemble reasoning, agent-native design.
- **6 thinking frameworks** in `prompts/`: code review, debugging, security audit, ship readiness, and more.

### 7. Triad Documentation System

- **3 templates** in `templates/`: VISION.md, SPEC.md, BUILDING.md.
- **6 doc templates** in `doc-templates/`: feature spec, bug report, ADR, deploy checklist, and more.
- **Self-installing**: `BUILDING-SETUP.md` bootstraps the Triad for any new project.

### 8. Workspace Generators

- **6 shell scripts** in `workspace/generators/`: new-project-workspace, new-task, new-feature, new-prep, new-bug, new-adr.
- **File-based organization**: Tasks, prep docs, scratch space, ideas, and decisions directories.
- **Bash 3.2+ compatible** for macOS.

### 9. Installation

- **Curl single file**: `curl -fsSL` the raw `squire.md` into project root or `~/.claude/CLAUDE.md`.
- **Full installer**: `./install.sh` with flags for `--dry-run`, `--commands-only`, `--uninstall`.
- **Cherry-pick**: Copy individual directories (`commands/`, `skills/`, `prompts/`) as needed.

### 10. Community Infrastructure

- **Contributor guidelines**: `CONTRIBUTING.md` covering what we accept, how to contribute, contribution standards, and code of conduct.
- **Issue templates**: `.github/ISSUE_TEMPLATE/` with bug report, feature request, and skill submission templates.
- **PR template**: `.github/PULL_REQUEST_TEMPLATE.md` with summary, testing, and checklist fields.
- **License**: MIT license in `LICENSE`.

### 11. Design System

- **Factory-Inspired**: All HTML artifacts generated by Squire skills follow the design language: near-black (#020202), near-white (#eeeeee), orange (#ef6f2e), amber (#f59e0b), teal (#4ecdc4).
- **Typography**: Geist + Geist Mono.
- **Constraints**: No shadows, gradients, or glow effects.

### 12. Multi-Model Orchestration

- **Director/Builder pattern**: Two AI models working together. Director (reasoning model) plans, reviews, integrates, ships. Builder (code generation model) executes scoped tasks.
- **Failure threshold**: 2 Builder failures on same task triggers Director takeover.

### System Role

Squire is the behavioral layer that sits between AI coding agents and the projects they work on. It does not interact with other systems at runtime. It changes agent behavior by being present in the agent's config or project context.

### Primary Actors

- `Developer` -- installs Squire, configures which components to use, benefits from improved agent behavior.
- `AI Agent` -- reads Squire files as context, applies behavioral rules, executes slash commands and skills.
- `Installer script` -- automates placement of files into the correct directories.

### Data Flow

```
Developer drops squire.md into project root or ~/.claude/
  -> AI Agent reads file as part of session context
  -> Behavioral rules constrain agent actions (preflight, scope, verification)
  -> Slash commands provide structured workflows on demand
  -> Skills inject domain expertise when invoked
  -> Agents delegate to specialized roles
  -> Pipeline enforces stage-gate discipline
```

## Current Boundaries

- Does NOT execute code itself: all execution happens through the AI agent's existing tools
- Does NOT have runtime dependencies, npm packages, or build steps
- Does NOT validate prompt quality or behavioral compliance automatically
- Does NOT support model-specific optimizations (rules are model-agnostic)
- Does NOT include a test suite for the prompts themselves
- Has contributor guidelines (`CONTRIBUTING.md`) and issue/PR templates, but no formal governance body or Discord
- Does NOT track adoption metrics beyond GitHub stars/forks
